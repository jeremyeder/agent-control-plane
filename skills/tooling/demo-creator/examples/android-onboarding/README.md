# Android onboarding example prerequisite

This scenario consumes the canonical Android build artifacts generated by Task
3 at:

- `components/mobile/dist/ambient-mobile.apk`
- `components/mobile/dist/ambient-mobile.apk.lock.json`

The pair is a required cross-task prerequisite. It is intentionally absent from
the isolated Task 4 demo-creator branch; do not add a fake APK, a fixture lock, or
alternate paths. Source-aware Doctor and the APK gate must validate the real
generated pair before any device mutation.
The lock must carry the exact source commit, tree, and `components/mobile`
path, with matching embedded commit/tree/schema metadata. Installation sends
the verified snapshot through an inherited descriptor; no APK pathname reaches
ADB or Package Manager.

The scenario authors `type: mobile` and no media source. Capture publishes the
portable `artifacts.mobileCapture`, `artifacts.pointerEvents`, and
`capture.source` inputs used by composition. The copied APK lock remains
validation provenance rather than a media input, preserving the same portable
composition boundary for a future iOS capture adapter.

`android-artoo-task3-contract.json` records the accepted Task 3 automation seam.
React Native exposes each `testID` verbatim as UIAutomator's `resource-id`, so
the scenario selects bare IDs such as `connection-origin`, `artoo-setup`, and
`artoo-composer`. It does not prefix the application ID or reuse human-readable
accessibility labels as automation identifiers. Pre-recording enters the owned
loopback origin and bearer token, signs in, and proves `artoo-setup` is present
before the secret-exposure audit. The recording then provisions memoryless
Artoo and submits one turn; it does not invent a Project field or a separate
client-owned connection flow.
